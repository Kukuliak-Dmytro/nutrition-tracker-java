## Завдання на курсову роботу

Розробити веб-орієнтований програмний застосунок для **особистого використання**, призначений для **локальної експлуатації** (без публічного деплою в інтернет), що забезпечує:

- ведення довідника інгредієнтів з поживною цінністю (калорії, білки, жири, вуглеводи на 100 г);
- створення та зберігання рецептів з прив’язкою до інгредієнтів та їх кількості;
- фіксацію історії приготувань (що і коли було приготовано) для аналізу та натхнення на подальше готування;
- підсумкову статистику та тижневі показники поживної цінності на основі історії;
- генерацію опису та покрокових інструкцій рецепта за допомогою штучного інтелекту за назвою та списком інгредієнтів.

Тип програмного рішення: веб-застосунок (клієнт у браузері, серверний API, локальна база даних).

## Вступ

Метою курсової роботи є розробка веб-застосунку для **особистого використання**, призначеного виключно для **локальної експлуатації** на комп’ютері користувача, без деплою в інтернет та без підтримки багатокористувацького режиму. Застосунок спрямований на відстеження харчування: керування калоріями, білками, жирами та вуглеводами, ведення рецептів та історії приготувань, щоб користувач міг бачити, що і коли він готував, та надихатися на подальше приготування страв. Для полегшення заповнення рецептів передбачено генерацію опису та покрокових інструкцій за допомогою штучного інтелекту за назвою страви та списком інгредієнтів.

Актуальність теми зумовлена зростанням інтересу до здорового харчування та домашнього приготування їжі, а також потребою в простому інструменті для особистого обліку без необхідності реєстрації в хмарних сервісах та передачі персональних даних на зовнішні сервери. Локальне використання забезпечує контроль над даними та мінімізує залежність від зовнішніх платформ.

У роботі реалізовано такі функціональні можливості: облік інгредієнтів з їхньою поживною цінністю на 100 г; створення та редагування рецептів з прив’язкою до інгредієнтів та кількості; історія приготувань із фіксацією дати та часу для аналізу та натхнення; підсумки по тижню за калоріями та БЖВ на основі історії; генерація опису та покрокових інструкцій рецепта за допомогою ШІ за назвою та списком інгредієнтів. Майбутнє застосування результатів роботи — використання на власному комп’ютері для планування та аналізу харчування та формування звички готувати вдома. Реалізація виконана сучасними програмними засобами: клієнтська частина на базі Next.js та React, серверна частина на Spring Boot (Java), база даних PostgreSQL; для генерації тексту рецептів використано API Google Gemini.

## 1. Постановка задачі

### 1.1. Тип програмного рішення

Програмний продукт є веб-орієнтованим застосунком архітектури «клієнт–сервер»: клієнтський інтерфейс працює в браузері, серверна частина реалізована у вигляді REST API, дані зберігаються в локальній реляційній базі даних. Застосунок розрахований **лише на локальне використання** (localhost): фронтенд та бекенд запускаються на машині користувача, публічний деплой у мережу Інтернет та мультикористувацька робота не передбачені.

### 1.2. Функціональні вимоги

З точки зору користувача до системи пред’являються такі функціональні вимоги:

- **Довідник інгредієнтів:** перегляд списку, пошук, фільтрація за категорією, пагінація; додавання, редагування та видалення інгредієнтів. Для кожного інгредієнта обов’язково вказуються назва, калорії, білки, жири та вуглеводи на 100 г, опційно — категорія.

- **Рецепти:** повний цикл CRUD. Рецепт містить назву, опис, покрокові інструкції, кількість порцій, час приготування, рейтинг та позначку «улюблений». До рецепта прив’язуються інгредієнти з кількістю у грамах; на основі цих даних автоматично обчислюється поживна цінність страви.

- **Історія приготувань:** фіксація події «коли приготовано який рецепт»; перегляд списку з курсорною пагінацією, пошуком за назвою рецепта, фільтрацією за діапазоном дат та сортуванням. Можливість видалення записів. Призначення — аналіз власних приготувань та натхнення на повторне готування страв.

- **Статистика та зведення:** кількість приготувань за період, серія днів поспіль (streak), тижнева підсумкова поживна цінність (калорії, БЖВ) на основі історії приготувань.

- **Генерація рецептів за допомогою ШІ:** формування опису та покрокових інструкцій рецепта за введеною назвою, кількістю порцій та списком інгредієнтів (з кількістю у грамах) шляхом виклику зовнішнього AI API для полегшення ведення рецептів.

### 1.3. Нефункціональні вимоги

Передбачається локальний запуск усіх компонентів: наявність Node.js та Java (JDK 21), Docker для підняття PostgreSQL. Інтерфейс повинен бути зручним для щоденного використання; дані зберігаються в БД; API повертає дані у форматі JSON; на стороні сервера застосовується валідація вхідних даних.

### 1.4. Обмеження

Система розрахована на одного користувача та один інстанс на одному комп’ютері; не призначена для промислового (production) розгортання в мережі та не реалізує авторизацію чи розмежування доступу між користувачами.

## 2. Огляд літератури

### 2.1. Предметна область

Здорове харчування та контроль над раціоном потребують систематичного обліку спожитих калорій та макронутрієнтів (білки, жири, вуглеводи) [1, 2]. Норми харчування та рекомендації щодо БЖВ містяться у джерелах з дієтології та харчування; для домашнього приготування важливим є не лише підрахунок калорій, а й збереження рецептів та історії приготувань, що сприяє формуванню звичок та повторному використанню перевірених страв [3]. Дослідження показують, що трекінг їжі та рефлексія над власними приготуваннями підвищують усвідомленість харчування [4].

### 2.2. Аналоги та прототипи

Існуючі комерційні та безкоштовні сервіси (MyFitnessPal, FatSecret, Yazio тощо) орієнтовані переважно на підрахунок калорій та харчування «на виході», часто з реєстрацією та зберіганням даних у хмарі [5, 6]. Шаблони для рецептів у Notion або інших інструментах не забезпечують автоматичного підрахунку поживної цінності на основі інгредієнтів. У даній роботі обрано підхід **локального особистого застосунку** з акцентом на рецепти та історію приготувань із можливістю генерації тексту рецептів за допомогою ШІ, що відрізняє проєкт від типових трекерів калорій та зберігає дані лише на машині користувача.

### 2.3. Варіанти технічної реалізації

Розглянуто варіанти: монолітний Next.js з API Routes та Prisma для простоти розгортання; розділена архітектура з окремим backend на Spring Boot для чіткого поділу відповідальностей та використання Java-екосистеми. Обрано варіант з Next.js на клієнті та Spring Boot на сервері: клієнт звертається до API по `/api`, проксування запитів на backend (порт 8080) налаштовується на рівні конфігурації Next.js. Як СУБД обрано PostgreSQL через надійність, підтримку UUID та зручність міграцій (Flyway). Для генерації тексту рецептів використано Google Gemini API (модель Gemini 2.5 Flash) через доступність, документацію та підтримку структурованого JSON у відповіді [7, 8].

### 2.4. Висновки розділу

Проєкт займає нішу особистого локального інструменту для обліку харчування та рецептів з історією приготувань та підтримкою ШІ. Вибір стеку (Next.js, Spring Boot, PostgreSQL, Gemini) забезпечує реалізацію всіх поставлених функціональних вимог та відповідає обмеженню на локальне використання без деплою.

## 3. Опис етапу проектування

### 3.1. Загальна архітектура системи

Система побудована за трирівневою архітектурою: рівень представлення (клієнт), рівень бізнес-логіки та API (сервер), рівень даних (БД). Клієнт — односторінковий застосунок на Next.js та React, що взаємодіє з сервером через HTTP REST API. Серверна частина реалізована на Spring Boot і обслуговує запити на порту 8080; база даних PostgreSQL працює у контейнері Docker на порту 5432. Деплой у зовнішню мережу не передбачено — усі компоненти запускаються на localhost. Схему взаємодії компонентів можна подати у вигляді діаграми компонентів (рис. 1 у додатках або в тексті).

### 3.2. Діаграма прецедентів (Use Case)

Єдиний актор — **Користувач**. Прецеденти: перегляд, додавання, редагування та видалення інгредієнтів; перегляд, додавання, редагування та видалення рецептів; додавання запису в історію приготувань та видалення запису; перегляд історії приготувань з фільтрами та сортуванням; перегляд статистики приготувань та тижневої поживної цінності; генерація опису та інструкцій рецепта за допомогою ШІ (за назвою та інгредієнтами). Діаграму прецедентів слід виконати в UML та підписати, наприклад: «Рис. 1. Діаграма прецедентів».

### 3.3. Діаграма компонентів

Компоненти: **Frontend** (Next.js, React) — відповідає за інтерфейс та відправку запитів до API; **Backend API** (Spring Boot) — обробка запитів, бізнес-логіка, доступ до БД та виклик AI API; **PostgreSQL** — зберігання даних; опційно окремо можна позначити підкомпонент інтеграції з AI (Gemini API). Зв’язки: клієнт → Backend по HTTP; Backend → PostgreSQL по JDBC; Backend → Gemini API по HTTPS. Діаграму компонентів розмістити з підписом «Рис. 2. Діаграма компонентів».

### 3.4. Модель даних (схема бази даних)

Модель даних складається з чотирьох основних таблиць. **ingredients:** id (UUID), name (унікальне), calories, protein, carbs, fat (на 100 г, DECIMAL 8,2), category, is_custom, created_at. **recipes:** id (UUID), name, description (TEXT), instructions (TEXT), servings, cooking_time, rating, is_favorite, created_at, updated_at. **recipe_ingredients:** id (UUID), recipe_id (FK → recipes, CASCADE), ingredient_id (FK → ingredients, CASCADE), quantity_grams (DECIMAL 8,2); унікальна пара (recipe_id, ingredient_id). **cooking_history:** id (UUID), recipe_id (FK → recipes, CASCADE), cooked_at, created_at. Індекси створені для полів пошуку та фільтрації (name, category, cooked_at тощо). Схему БД доцільно подати у вигляді ER-діаграми або схеми таблиць з підписом «Рис. 3. Схема бази даних»; при великому розмірі — у додатку на альбомній сторінці.

### 3.5. Діаграми класів (ключові підсистеми)

На етапі проектування доцільно відобразити ключові класи backend: сутності JPA (Ingredient, Recipe, RecipeIngredient, CookingHistory), основні сервіси (CookingHistoryService, NutritionSummaryService, CookingStatsService, RecipeAiService, RecipeService, IngredientService), контролери (IngredientController, RecipeController, CookingHistoryController, RecipeAiController, DashboardController) та їх зв’язки. Діаграму класів можна розбити на фрагменти (наприклад, модель даних та сервісний шар) і підписати відповідно (Рис. 4, 5).

## 4. Програмне рішення

### 4.1. Технологічний стек

Клієнт: Next.js 15, React 19, Tailwind CSS, TanStack Query (React Query), Axios, react-hook-form, Zod для валідації, Recharts для графіків, next-intl для локалізації. Сервер: Spring Boot 3.4, Java 21, Spring Data JPA, Hibernate, Flyway для міграцій, Jakarta Bean Validation. База даних: PostgreSQL 16 у Docker-контейнері. Для генерації рецептів використано Google Gemini API (модель Gemini 2.5 Flash). Усі зовнішні бібліотеки та API використані згідно з умовами ліцензій; можливість їх використання узгоджена з керівником роботи.

### 4.2. Реалізація API

API має префікс `/api`. Групи endpoint: **інгредієнти** — GET/POST `/api/ingredients`, GET/PATCH/DELETE `/api/ingredients/{id}` (пошук, фільтр, пагінація); **рецепти** — GET/POST `/api/recipes`, GET/PATCH/DELETE `/api/recipes/{id}` (пошук, фільтр, сортування); **історія приготувань** — GET/POST `/api/cooking-history`, DELETE `/api/cooking-history/{id}`, GET `/api/cooking-history/stats`, GET `/api/cooking-history/nutrition`; **дашборд** — GET `/api/dashboard`; **AI** — POST `/api/ai/generate-recipe`, GET `/api/ai/generate-recipe` (перевірка доступності сервісу). Контролери приймають DTO, делегують логіку сервісам, повертають ResponseEntity з відповідними DTO. Приклад: CookingHistoryController обробляє параметри запиту (cursor, limit, search, startDate, endDate, sortBy, sortOrder), формує CookingHistoryQueryParams та викликає CookingHistoryService, CookingStatsService та NutritionSummaryService для відповідних endpoint.

### 4.3. Робота з базою даних

Сутності JPA відповідають таблицям БД; зв’язки описані через @OneToMany, @ManyToOne. Репозиторії (інтерфейси JpaRepository) надають методи для вибірки з підтримкою пагінації та специфікацій за потребою. Міграції Flyway (V1__create_schema.sql) створюють таблиці та індекси при першому запуску. Поживна цінність рецепта обчислюється на основі суми (quantity_grams/100 * значення на 100 г) по всіх інгредієнтах рецепта; тижневі підсумки формуються з записів cooking_history за поточний тиждень з урахуванням порцій та нормованої поживної цінності на одну порцію.

### 4.4. Генерація рецептів за допомогою ШІ

Сервіс RecipeAiService приймає назву рецепта, кількість порцій та список інгредієнтів (назва, кількість у грамах). Формується текст запиту (prompt) для моделі Gemini з проханням повернути JSON з полями description (короткий опис) та instructions (масив кроків). Відповідь парситься; instructions можуть приходити масивом або рядком — обробка приведена до єдиного текстового формату для збереження в полі instructions рецепта. API-ключ Gemini зберігається лише на сервері (змінна оточення або application.yml), що виключає його потрапляння в клієнтський код.

### 4.5. Інтерфейс користувача

Структура сторінок: головна (дашборд) з останніми рецептами, останньою історією приготувань та тижневою поживною цінністю; сторінка інгредієнтів (список, додавання/редагування); сторінка рецептів (список, деталі, створення/редагування з вибором інгредієнтів та кількості); форма рецепта містить кнопку/функцію генерації опису та інструкцій за назвою та інгредієнтами через виклик `/api/ai/generate-recipe`; сторінка історії приготувань з фільтрами та можливістю додати запис «приготовано». Для роботи з API використовуються хуки TanStack Query; форми — react-hook-form та Zod; компоненти UI побудовані з використанням Radix UI та Tailwind CSS.

## 5. Опис проведених експериментів

### 5.1. Умови проведення

Експерименти проводилися на локальній машині з встановленими Node.js (20+), Java 21 (JDK), Maven, Docker Desktop. База даних PostgreSQL піднімалася командою `docker compose up -d postgres` (або `npm run setup:db`). У конфігурації backend (application.yml або змінні оточення) вказано DATABASE_URL (JDBC URL) та опційно GEMINI_API_KEY для AI. Backend запускався на порту 8080, frontend — на порту 3000; у next.config налаштовано rewrite запитів з `/api` на `http://localhost:8080/api`.

### 5.2. Інструкція для користувача

Відкрити в браузері адресу http://localhost:3000. На головній сторінці відображаються останні рецепти, останні записи історії приготувань та зведення по тижню. Додати інгредієнт: перейти в розділ інгредієнтів, натиснути кнопку додавання, заповнити назву та поживну цінність на 100 г. Створити рецепт: перейти в рецепти, додати новий, вказати назву, порції, додати інгредієнти з кількістю; за бажанням скористатися кнопкою генерації опису та інструкцій (ввести назву та інгредієнти, отримати текст від ШІ). Зафіксувати приготування: з картки рецепта або зі сторінки історії додати запис «приготовано» з датою/часом. Переглянути історію та тижневі підсумки можна на дашборді та на відповідній сторінці історії приготувань.

### 5.3. Інструкція для адміністратора (розгортання на новому комп’ютері)

Встановити залежності: Node.js, JDK 21, Maven (або використовувати Maven Wrapper у проєкті), Docker. Клонувати репозиторій; у корені проєкту виконати `npm run setup:db` (або `docker compose up -d postgres`) для підняття БД. Налаштувати application.yml (або .env) з правильним JDBC URL та, за потреби, GEMINI_API_KEY. Запустити backend: `npm run backend:dev` або `./mvnw spring-boot:run` у каталозі backend. При першому запуску Flyway виконає міграції та, за налаштуванням, seed-дані. Встановити залежності frontend: `npm run frontend:install` (або `cd frontend && npm install`). Запустити frontend: `npm run frontend:dev`. Для зручності можна використовувати `npm run dev` для одночасного запуску БД, backend та frontend за наявності відповідних скриптів. Деплой у зовнішню мережу не передбачено.

### 5.4. Екранограми

Рекомендується включити 3–6 знімків екрану з підписами: головна сторінка (дашборд); список рецептів або сторінка деталей рецепта; форма створення/редагування рецепта з кнопкою генерації ШІ; сторінка історії приготувань; перегляд тижневої поживної цінності або статистики. Великі екранограми можна винести в додатки. Підписи: «Рис. 6. Головна сторінка (дашборд)», «Рис. 7. Історія приготувань» тощо.

## Висновки

У курсовій роботі реалізовано веб-застосунок для особистого обліку харчування та рецептів, призначений для локального використання без публічного деплою. Досягнуто повний цикл CRUD для інгредієнтів та рецептів, реалізовано історію приготувань з фільтрацією та сортуванням, статистику приготувань та тижневі підсумки поживної цінності (калорії, БЖВ), а також інтеграцію з AI API для генерації опису та покрокових інструкцій рецепта за назвою та списком інгредієнтів.

У процесі виконання роботи набуті практичні навички розробки повноцінного веб-застосунку: побудова клієнтського інтерфейсу на Next.js та React, проектування та реалізація REST API на Spring Boot, робота з PostgreSQL та JPA, використання міграцій Flyway, інтеграція зовнішніх API (Google Gemini). Перспективами розвитку системи можуть бути розширення довідника інгредієнтів, додаткові фільтри та звіти в історії приготувань, експорт даних у файл, а за бажанням — опційна підтримка кількох профілів (користувачів) при збереженні орієнтації на локальне використання.
